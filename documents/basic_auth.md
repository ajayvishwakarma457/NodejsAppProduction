# Basic Authentication Reference Guide

This document describes how basic authentication is implemented, covering password hashing, JSON Web Tokens (JWT), and contrasting cookie-based session authentication with token-based authentication.

---

## 1. Password Hashing (bcryptjs)

Passwords should **never** be stored in plain text inside a database. If the database is compromised, all user passwords will be leaked.

### Hashing vs Encryption
* **Encryption**: A two-way function. You encrypt plaintext with a key, and you can decrypt it back using the same/another key.
* **Hashing**: A one-way function. Plaintext is put through an algorithm to generate a fixed-length string (the hash) which **cannot** be reversed.

### How bcrypt Works
`bcrypt` is designed to be slow and computationally expensive (to resist brute-force attacks).
1. **Salt**: A random string added to the password before hashing. It prevents attackers from using pre-calculated lists (Rainbow Tables) to look up hashed passwords.
2. **Work Factor (Cost)**: Determines how many times the hashing function runs. In production, a cost of `10` or higher is standard.

```javascript
const bcrypt = require('bcryptjs');

// 1. Generate salt and hash
const salt = await bcrypt.genSalt(10);
const hash = await bcrypt.hash("userPassword123", salt);

// 2. Verify password
const isMatch = await bcrypt.compare("userInputPassword", hash); // returns true/false
```

## 2. JSON Web Tokens (JWT)

A JSON Web Token is an open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object.

### Structure of a JWT:
A token is split by periods (`.`) into three parts:
1. **Header**: Contains the signing algorithm (e.g. `HS256`) and the token type (`JWT`).
2. **Payload**: Contains the claims (user data such as ID, role, and expiration timestamp).
3. **Signature**: Cryptographic signature generated by taking the encoded Header + Payload and signing them with a secret key. This proves the token hasn't been tampered with.

```text
header.payload.signature
```

### JWT Execution in Node.js
```javascript
const jwt = require('jsonwebtoken');
const SECRET = process.env.JWT_SECRET;

// 1. Signing (creating a token)
const token = jwt.sign(
  { id: 42, role: 'admin' }, 
  SECRET, 
  { expiresIn: '1h' } // Expiry security
);

// 2. Verification
try {
  const decoded = jwt.verify(token, SECRET);
  console.log(decoded.id); // 42
} catch (err) {
  // TokenExpiredError or JsonWebTokenError
}
```

## 3. Cookie-Based Sessions vs Token-Based Auth

Choosing how to store session state is a critical architectural decision.

| Comparison Metric | Cookie-Based Session Auth | Token-Based Auth (JWT) |
|---|---|---|
| **State Storage** | **Stateful**: Server stores session ID in memory/DB (e.g. Redis). Client stores only session ID in a cookie. | **Stateless**: Server stores nothing. The token itself contains all the state (decoded payload). |
| **Verification** | Server must query the DB/Redis for every request to check session validity. | Server decodes the token cryptographically on the CPU (no DB lookup needed). |
| **Scalability** | Harder to scale horizontally (requires sharing session state across multiple servers). | Highly scalable. Any server with the signing secret key can verify the token. |
| **Security Risks** | Vulnerable to **CSRF (Cross-Site Request Forgery)**. Protect by setting `SameSite: strict` cookies. | Vulnerable to **XSS (Cross-Site Scripting)** if stored in `localStorage`. |
| **Revocation** | Easy. Simply delete the session entry from the DB/Redis. | Harder. The token is valid until it expires, unless you implement a blacklist system (using Redis). |
